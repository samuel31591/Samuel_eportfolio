    Compromising a Medical Mannequin.

    It has been observed that the pace of technological advancement in the medical industry is faster than the pace of cyber security considerations (Glisson et al., 2015). The critical observation is that a breach of a training tool would result in a clinician receiving incorrect guidance when problem-solving is insightful. A discrepancy between technology and security considerations in this industry can potentially result in a fatality.

    The studies were conducted on a mannequin model (istan), a wireless patient simulator used at the university's College of Nursing. The device can bleed, secrete bodily fluids, has blood pressure and heart rate, and breathes realistically. The simulator links with iStan software, which controls the mannequin remotely (Still, 2015). By identifying the network security solution and network protocol as the vulnerable components, brute-force attacks against the router PIN, as well as denial-of-service attacks using open-source tools such as Backtrack

    The tools implemented in this experiment were not sophisticated or difficult to acquire. However, this alone highlights the lack of security currently being implemented in these devices.

    A brute force attack that targeted the Wi-Fi and a denial-of-service attack rendering the unit inoperable were used to demonstrate the vulnerability of the medical mannequin. A Denial of service (DOS) attack is designed to shut down or make the target unusable. This is accomplished by flooding the target with a large amount of data that the device cannot handle, causing it to become unresponsive.

    There are several ways to mitigate brute force attacks, such as changing passwords frequently, increasing password length, limiting login attempts, and implementing multifactor authentication. A proactive approach would be setting up alerts for brute force attacks and monitoring and stopping them before they are fully deployed (Varonis, 2021).

    DOS attacks can be prevented by proactively monitoring a system for any unusual activity by creating a baseline. In addition to this monitoring, an effective method could also be to implement a VPN with MFA required to access the medical device (Phoenixnapp,2018). Over the years, medical devices have gone from being utterly non-connected to being both wirelessly and software controlled. It's worth bearing in mind that there is an international standard to which medical devices must certify; The IEC 62304 Medical device software, which was put in place to prevent cyber-criminal events on medical devices. However, the standard has not been updated to consider and include either network connectivity or cybersecurity (Williams, Woolward, 2015). When medical device manufacturers are required by law to implement strict security measures into the lifecycle of their products, we will likely see the threats above, and vulnerabilities decrease.

    By disabling WPS, the mannequin can't easily connect to the network, making it more vulnerable. The WiFi Protected Setup (WPS) is a wireless network security standard that makes connections between a router and wireless devices (mannequins) faster and easier. Wireless Protected Setup is a key exchange mechanism rather than a security mechanism. It facilitates connections between devices and access points and is one of the vulnerable points of a network.

    WPS works with WPA/WPA2/PSK and allows devices to establish connections to WLANs. Utilizing the following methods

  • PIN: Entering a PIN on the device

  • Push-button: A physical or virtual push-button should be pressed on both the access point and the client device during the connection phase.

  • NFC: To use Near Field Communication, the client must bring their device close to the access point.

  • USB drive: Necessary connection information exchange between an access point and a device is done using a USB drive.

    • The short length of WPS PINs and the fact that both PIN parts are validated separately means that an online brute force attack on the PIN can be made within several hours, allowing an attacker to access a WLAN (mannequin device). Wireless LAN clients can authenticate their users and ensure user confidentiality through WPA or WPA2. WPA is vulnerable to the offline dictionary attack since the information required to create and verify a session key is broadcasted (Noureddine Boudriga, 2019, p.270) Using WPA2/enterprise security and mutual client and server authentication provides the most secure and protected wireless network compared to the WPA and WEP.

    Both models have advantages and disadvantages. The OSI model should be preferred over the TCP/IP approach if appropriate documentation, specification, and modularisation are essential. However, if the network's implementation, stability, and security are more critical, the TCP/IP model should be implemented

    SIMILARITIES BETWEEN THE OSI AND TCP/IP MODELS.

    1.Reference Model: Both the OSI and the TCP/IP are reference models.

    2.Layered Architecture: Both the OSI and TCP/IP models have a layered architecture, with the OSI model having7 layers while the TCP/IP has five layers.

    3.Each layer provides different functionalities in the network.

    4.Protocols: Both the OSI and the TCP/IP model use different protocols in different layers for the proper implementation of the model over the network.

    5.Functionalities: The OSI and the TCP/IP model layers provide approximately the same functionality.

    DIFFERENCES BETWEEN THE OSI AND THE TCP/IP MODEL

    1.Evolution: The OSI model evolved as a logical and conceptual model. The TCP/IP model is implemented first with the specified protocols and then documented. So, if someone needs the theoretical aspects of the model, they should go with the OSI model. But if someone wants to implement the model practically - use the TCP-IP model.

    2.Objective: The OSI model's goal is to create a generic standard for describing connection methods, layered architecture, services, interfaces, and protocols. The TCP/IP model, on the other hand, strives to provide a secure and end-to-end transmission model. The OSI model should be used if a generic and standard model is required. However, the TCP/IP paradigm should be chosen if network stability and security are needed.

    3.Area Focused: The OSI model is a generic model that is hence universal. It can be utilised in various sorts of networks according to the standards. The TCP/IP model, on the other hand, is protocol-based and compatible with existing Internet architecture. The OSI Model should be used if someone needs a universal model that can be applied to various networks.

    4.Documentation: The OSI model specifies the three central concepts of services, interfaces, and protocols. The TCP/IP model is not so clear. So, if someone needs documentation and guidance while implementing the network, they should refer to the OSI model.

    5.Set-up and Configuration: The OSI model is simple to set up and configure because it is a generic model. The TCP/IP approach, on the other hand, is complicated since it is only compatible with certain types of networks. So, the OSI model is superior when it comes to network setup and configuration.

    6.Modularity: If someone focuses on a more modular network with proper functionalities, they should go for the TCP/IP model. Both models are modular, but the OSI model has more layers (7) than the TCP-IP model(5 layers).

    7.Replacing Protocols: The OSI model is a protocol-independent model which allows us to implement our protocols as per our needs. On the other hand, the TCP/IP model is protocol dependent and defines a specific set of protocols for implementing the model. It is very complex to make changes or replace some protocols in the TCP / IP model. So, if someone needs the specified set of protocols, they should go with the original model.

    8.Data Delivery: In the OSI model, the transport layer facilitates the connection-oriented transfer, and hence it guarantees the delivery of packets. Data delivery is the functionality of the Transport layer in both models. So, we can use the OSI model if we want to guarantee the proper data delivery over the network.

    9.Reliable and Secure Connection: The OSI model does not have any special mechanism for providing a reliable and secure connection for data transmission. On the other hand, the TCP/IP model has a 3-way handshake mechanism to provide a safe and dependable connection link.

The data controller must demonstrate that to be exempted from publishing personal data under section 22A of the ACT. Personal data that is processed only for the purposes specified in this subsection is exempt if:

    (a)the processing is performed to publish any journalistic, literary, or artistic material by a person,

    (b) considers that the public interest in freedom of expression is of particular importance, and the data controller reasonably believes publication would be in the public interest.

    (c) based on all the circumstances, the controller believes compliance with that provision is incompatible with the purposes (The National Archives, 2022).

    This case shows that section 22A of the Acts does not grant absolute immunity that can be routinely invoked to justify publishing unnecessary personal data by publishers and journalists alike.     1.0 Introduction

    Website security is vital to any website, and organisations must ensure they implement appropriate levels of security when designing them. All websites that will process or store European or UK personal information must be compliant with the General Data Protection Regulation (GDPR) and ensure appropriate organisational and technical measures are put in place to ensure the security of this data (GDPR, 2018). This report uses recommended vulnerability scanning tools to assess the website for the potential weaknesses (Daud, 2015) discussed in the design document.

    2.0 Scanning Tools

    Vulnerability scanning can be conducted through manual testing or available tools (Daud, 2014). For the scanning performed on allmytype.co.uk, several tools were reviewed and selected based on criteria that assessed the tools' ease of use, flexibility, reputation, licensing, and ease of installation.

    2.1 Kali Linux

    Kali Linux can be installed as a virtual machine on a device allowing users access to multiple applications and frameworks (Bhatt, 2018). While some of the tools offered as part of the package are more complex and suited to experienced penetration testers, the variety available means there is also an offering for beginners with little knowledge of Kali or security testing (Leroux, 2020), making the tool as straightforward or as complex as the user requires. While downloading Kali Linux from their website is not complicated, the user must first have some virtualisation software to install Kali on, making the process more complex. For scanning allmytype.co.uk, Kali Linux was chosen to gain access to a range of tools.Through Kali Linux we utilised Nikto, a command-line vulnerability scanner that scans for both security vulnerabilities and misconfigured web servers. We combined this with Network Mapper (Nmap) a free open-source tool (Kaur & Kaur, 2017), often used for network discovery and auditing by using IP packets to determine what hosts are available (Bhingardeve & Franklin, 2018).

    3.1 Broken Access Control

    GDPR (2018) requires allmytype.co.uk to protect the integrity confidentiality and availability of personal data, access controls must be implemented to manage who has access to the data in allmytype.co.uk. Broken access control is up from fifth place in OWASP due to a number of notable Common Weakness Enumeration (CWEs). Information Disclosure to Unauthorised Personnel, GWE-201: Inserting Sensitive Data into Sent Data, and GWE-352: Access to Sent Data by Unauthorised Personnel. Cross-Site Request Forgery (CSRF). A scan of allmytype.co.uk showed the presence of the CSRF vulnerability which allows an attacker to force a user's web browser to conduct malicious activity on the site (Siddiqui & Verma, 2011).

    These vulnerabilities include:

  • Access should only be allowed for specific capabilities, roles, or users, but is open to everybody in violation of the concept of least privilege or deny by default.

  • Modifying the URL (parameter tampering or force browsing) or internal application state, or using an attack tool to manipulate API requests, to get around access control checks.

  • By giving a unique identity, you can read or edit someone else's account

    • 3.1.2 Mitigations

  • Deny access by default.

  • Log access control failures, alert admins when appropriate for example, repeated failed login attempts.

  • Implement access control mechanisms once and re-use them throughout the application. Access controls should enforce ownership rather than accept that the user is permitted to create, read, update, or delete any record.

  • Disable directory listing for the web server and ensure that backup files and file metadata are not present within the web root.

  • Limit API and controller access to minimise the harm from automated attack tooling.

  • After logging out, stateful session identifiers should be invalidated. Tokens based on stateless JWT are recommended to be short-lived in order to minimise the window of opportunity for an attacker. The OAuth revoke access standards should be followed for longer-lived JWTs.

    • 3.2 Cryptographic Failures

    Article 32 of GDPR suggests using appropriate encryption to secure any personal information being stored or while in transit (GDPR, 2018). A Nikto scan of allmytype.co.uk revealed the Transport Layer Security (TLS) certificate is encrypted, as shown in figure 5, using AEAD Advanced Encryption Standard (AES) with 256bit key in Galois/Counter mode (AES 256 GCM) and Secure Hash Algorithm 384 (SHA384) which is more secure than other algorithms like SHA-1 (TECHCOMMUNITY.MICROSOFT.COM, 2021).

    AES encryption encrypts plaintext inputs and cryptographic keys and yields an encrypted block of byte data that is decrypted by the receiver with the same key. This AES encryption supports the block cipher mode GCM which uses a cryptographic key of 128 or 256 bits in length (Google Cloud, N.D).

    Security Hash Algorithm 2 (SHA384) produces a hash value of 384 bits as a 96-digit hexadecimal number. This helps verify if a file has been altered (WEB Tools, N.D).

    Based on the above, the site is not likely to be vulnerable to SSL TLS attacks such as Advanced Persistent Malware, SSL Stripping Attacks, Man-in-the-Middle attacks, and brute force attacks and AES is known to be more resistant to the recent quantum attacks, which makes it a preferred encryption solution (Rao et al., 2017).

Misconfigurations, default accounts, unpatched flaws, unused access files, directories, and unprotected files, along with attackers' tactics to gain control of your systems, can be detected by automatic scanners.     The website might be vulnerable if;

  • There are no or incorrectly specified security configurations in any section of the application stack.

  • Features that aren't required are enabled or installed (open ports, services, pages, accounts, or privileges)

  • usage of default passwords and accounts

  • Error handling that reveals stack traces or other data which can provide useful information to an attacker

  • The most recent security settings are either disabled or improperly configured.

  • Application servers, application frameworks (e.g., Struts, Spring, ASP.NET), libraries, and databases do not have their security settings configured to safe values.

  • Software is updated or susceptible; systems are more vulnerable without a systematic application security configuration process.

    • 3.3.1 Mitigations

    Installation processes that are secure should be implemented, including:

  • Establish a repeatable hardening procedure that can be implemented fast and efficiently in a secure environment. All environments should be established up the same way, with different credentials for each domain which should be automated to save time.

  • Retain a platform simple by eliminating unnecessary features, components, documentation, and samples. To minimise the attack surface, remove or do not install new features and frameworks that are not required.

  • Implement a Patch Management process to review required updates and configuration changes.

  • implementation of segmentation, containerization, or cloud security groups, as segmented application design allows effective and secure separation between components or tenants (ACLs).

  • To ensure the effectiveness of setups and settings across all contexts, an automated procedure should be used.

    • 3.4 Vulnerable and Outdated Components

    One of the top application security threats that web developers must address, according to the OWASP 2021, is vulnerable and obsolete components (OWASP, 2021).This includes using out of support operation systems, databases, and applications as part of the website development and not ensuring identified vulnerabilities are patched.

    The Nikto scan (Figure 5) showed the website is running on an Apache server and uses the Imunify360 web server security suite. Due to being open source and multi-platform (Vlad, 2022), Apache is one of the most popular choices of web server. In 2019, an update was released to patch CVE-2019-0211, which gave attackers full root access to the server (Narang, 2019). In 2022, Apache released an update to address CVE-2022-23943, a denial-of-service vulnerability (Moussalli, 2022). The release of updates must be kept up with so as not to become susceptible to a breach through a known vulnerability and ensure there is an appropriate process in place to alert the operations team for critical patches so as not to cause a delay (Luszcz, 2018).

    Imunify360 offers website protection and web server security capabilities, including intrusion prevention and detection systems, network firewalls, antivirus protection, and patch management. Allmytype.co.uk uses Imunify 360, a security platform designed for web hosting servers. Researchers from Cisco's Talos unit discovered that Imunify360 could exploit a high-severity vulnerability. A remote attacker can execute arbitrary code on a server using a specially crafted file. This has been reported to the vendor and is now patched in a new version, 5.11.3 (Kovacs,2021.3).

    3.4.1 Mitigations

  • Although the Apache version being used is unclear, the organisation must make sure it is patched to the latest version (2.4.46), which includes some security fixes.

  • Update with a version of Imunify360 at least 5.11.3 or higher.

    • 3.5 Cross-Site Scripting (XSS)

    The scan in figure 5 shows The X-XSS protection header is not defined. Data is sent to a web user as dynamic content without being validated to verify that it doesn't contain malicious software. XSS attacks occur when an untrusted source like a web browser enters data into a web application without validation. Web browsers typically receive malicious code in the form of JavaScript, but it can also be HTML, Flash, or any other code the browser can execute.

    Attacks using XSS are almost limitless, but they commonly involve sending private information to the attacker and redirecting it back to the victim. An XSS attacker may use the victim to access content controlled by the attacker or perform other malicious operations on the victim's machine while posing as the infected site itself.

    Figure 5 - Nikto scan

    3.5.1 Mitigations

    Effective mitigation for XSS vulnerabilities will require several effective strategies including;

  • 1. Filter input on arrival.

  • 2. Encode data on output at the point where user-controllable data is output in HTTP responses, encode the output to prevent it from being interpreted as active content.

  • 3. Use proper heards to avoid XSS in answers that aren't supposed to contain any HTML or JavaScript.
